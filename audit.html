Audit Cycle

How we found and fixed 20+ unfiltered queries, prevented future data leakage, and established continuous enforcement — the story of turning a potential integrity failure into a systematic safeguard.

Background

The OTM pipeline was built in stages over several months. Stage 1 (SAR processing) was written first, followed by Stage 2 (market data + ML models), then Stage 3 (signal generation + paper trading). Each stage was developed and tested independently before integration.

The out-of-sample holdout policy (cutoff: May 1, 2025) was formalized after the pipeline was operational. This meant that queries written during initial development did not consistently include date filters — they were written to process all available data, not to respect a training/evaluation boundary that didn't exist yet.

The Audit

A static analysis pass scanned the entire codebase for queries that reference any of 19 known training tables. The scanner used two methods:

  1. SQL file scan — every .sql file checked for training table references without date filters
  2. Python AST parse — every .py file's AST was walked to extract SQL strings from string literals, f-strings (JoinedStr nodes), and triple-quoted blocks

Key Finding: Prevention, Not Correction

The audit identified queries that would have caused leakage once training tables were populated. At the time of audit, the derived training tables (hub_state_day, linkage_features) contained 0 rows — they had never been backfilled.

Consequently, no model had ever been trained on leaked data, because no model had ever been trained on these tables at all. The audit prevented future leakage rather than correcting historical contamination.

All models reported in this project were trained from scratch with the holdout filter in place from the first run.

Fixes Applied

  1. Added WHERE date < '2025-05-01' to all 20+ Category A queries
  2. Threaded query_ts parameter through all 8 memory retrieval paths (Category B)
  3. Created @oos-eval-only pragma system for Category C evaluation code
  4. Registered Category D real-time files in NO_DATE_FILTER_NEEDED allowlist
  5. Built bq_safe.py runtime enforcement as defence-in-depth
  6. Added 5 CI-blocking tests (test_oos_leakage.py)
  7. Added 16 causal leakage tests (test_causal_leakage.py)

Continuous Enforcement

The audit was a one-time event; the enforcement is permanent. Every future code change goes through:

  • Static analysis (CI) — 5 tests scan every commit for training table access without filters
  • Runtime enforcement — TrainingBQClient raises OOSLeakageError on dynamically constructed queries
  • Causal leakage tests — 16 tests verify query_ts threading through all memory paths
  • Allowlist validation — stale allowlist entries are caught automatically

No query can access training data in the OOS window without being caught at build time or runtime.

Lessons Learned

1. Audit early, even when you think it's fine

The fact that training tables had 0 rows was lucky timing. If backfill had run before the audit, we would have needed to retrain everything. Run the audit before the data exists, not after.

2. Static analysis catches what code review misses

A human reviewer might miss a date filter buried in a 50-line SQL string. AST parsing of every Python file catches it mechanically.

3. Defence-in-depth matters

Static analysis can miss dynamically constructed queries. Runtime enforcement (bq_safe.py) catches those. Neither alone is sufficient.

4. Make integrity controls visible

Publishing the audit cycle, including the finding of 20+ violations, builds more trust than claiming "no issues found." Honest reporting of the fix process is more credible than claiming perfection.
